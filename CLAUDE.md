## このプロジェクトは何か

**「zu — 伝わりやすい形について」** という、視覚を通じた情報伝達の Tips 集 Web サイト。

核となる再定義は **「紙の上に文字を置いた時点で、それはもう図だ」**。スライド・資料・Excel・ホワイトボード・チャット投稿まで、見せて伝えるものすべてを〈図〉と総称する。「インフォグラフィック」のような借り物の用語は使わず、サイト自身が〈図〉と再定義する。

思想：作者（craftpaperbag）は長年、資料作成・Web デザインのノウハウを「溜め込むこと」が価値だと思ってきた。だが溜め込んだ知識は使われなければ消える。このサイトは溜め込んだものを全部・わかりやすく開示し、「人が橋をかける（伝える）ための、支えの一石」になることを目的とする。

内容の特徴：見た目のコツだけでなく、**説得力のある現実世界の観察や例え**とともに伝える。図が6〜7割、文字が3〜4割。難しいデザイン・認知心理学の専門用語は避け、書くとしても末尾に小さく添える。トーンは平易・ストレート・本質的・簡潔だが丁寧に。

## 技術・配信の前提

- **ビルドなし・単体配信**で完結させる。`index.html` を GitHub Pages 直下で公開する想定。
- データは2つの外部 JS ファイルに分離してある。どちらも `<script src>` で先に読み込み、`index.html` 側の描画ロジックが参照する（バンドラ不要）。
  - `tips.js` — Tipsのたね（`TIPS_space` などカテゴリ別の `TIPS_*` 配列）。**種を増やすときはここの該当配列に1ブロック足すだけ。** 1本につなぐ `const TIPS` の連結処理は `index.html` 側にある。
  - `history.js` — 更新履歴（`HISTORY` 配列）。見える変更のたびに先頭へ追記する。
  - 構造・スタイル・描画ロジック（`CATEGORIES`・`index.html` の `<script>`）はあまり変わらないので `index.html` に残す。
  - ※さらにデータが増えても CSS/JS の全面3分割はしない。編集の主戦場（履歴・たね）だけ外に出す方針で、単体配信のシンプルさを守る。
- Tailwind（CDN）+ lucide アイコン（CDN）。
- **固定画像（PNG/JPG）は使わない。図はすべて SVG または HTML 要素で直接描く。**
- スクショを撮るときは `SCREENSHOTS.md` を見る（クラウド環境は CDN が egress で遮断されるため、Tailwind ローカルビルド＋puppeteer で本番同等に撮る手順を残してある）。
- ライセンスは留保（All Rights Reserved）。
- データとビューを分離する。**種を増やすときは配列に1ブロック足すだけ**で済む構造を壊さないこと。

## カテゴリ体系（新6枠）

1種 = **1主カテゴリ + 複数タグ**。複数カテゴリへの所属は許さず、横串はタグで張る。

| ID | ラベル | lucideアイコン |
|---|---|---|
| `space` | 余白と配置 | `move-diagonal` |
| `hierarchy` | 強弱をつける | `arrow-up-narrow-wide` |
| `color` | 色と濃淡 | `palette` |
| `text` | 文字を図として | `type` |
| `draw` | 線・矢印・グラフ | `spline` |
| `flow` | 進め方 | `route` |

> `flow`（進め方）は、図そのものの作法ではなく、図を作る**過程・こころがけ**を扱う枠。「早めに見せる」「下書きから」など、視覚の話に収まらない工程系の種はここに置く。

> 注意：プロトタイプの初期 HTML には旧5枠（align / shape / color / text / cut）が残っている可能性がある。見つけたらこの新5枠に更新し、既存 Tips の `cat` も新 ID に付け替える。

## 1エントリのスキーマ

```js
{
  id,       // 一意の識別子（kebab-case）
  cat,      // 主カテゴリID（上表のいずれか）
  tags,     // 横串タグの配列
  title,    // 「角丸は、安心する」のような短い言い切り
  claim,    // 一文の主張（本文色で表示。朱は図の中の「ここぞの一点」だけに使う）
  why,      // 現実世界の観察・例え（このサイトの命）
  visual,   // SVG/HTML 直描きの図（主役・6〜7割）
  apply,    // 明日から使える具体
  term,     // 任意。末尾に小さく添える専門用語。なければ空文字
  added,    // 追加日（"YYYY-MM-DD"）。種を足した日を入れる。新着順の並べ替えと、詳細・新着順カードの日付表示に使う
  links,    // 任意。他の覚え書きへのつながり（後述）。[{to:"id", rel:"echo|akin|tension|then"}]
}
```

`why` と `visual` がこのサイトの命。普通の Tips 集は `apply` しかないが、zu は **「なぜそう感じるか」を現実世界で説明する**ことが差別化点。

### why を書くときの作法（重要）

- 誰もが知っている**身体感覚・日常の場面**に接続する。専門知に頼らない。
- **値段・権威・経済格差に逃げない。**（例：「一流店／安い店」のような対比は使わない。体験そのもので語る。）
- 良い側だけ言い切り、良し悪しの対比は図に任せる。文章で対比を繰り返さない（それ自体が「増やすほど薄まる」の実演になる）。
- 一文で主張を言えること。`aria-label` がその自己検算になる。

### 覚え書き同士の「つながり」（links）

種は単独で立つが、互いに響き合う。`links` で関連する覚え書きへ橋を架ける。詳細モーダルの最下部に「つながり」節として出る。

- **片側だけ書く。逆向きは自動で出る。** A に `{to:"B", rel:...}` と書けば、B のページにも逆リンクが自動で現れる。両端に書かない（不整合の元）。「1ブロック足すだけ」の原則をリンクにも貫く。
- **rel は4種**。色は足さず、アイコンと静かな言葉だけで種別を分ける（サイト自身が説く「色を増やさない」「`semantic-color`」を濁さないため）。
  - `echo` — 別の分野・側面で**同じこと**を言っている（対称）。例：`dilute`(増やすほど薄まる)↔`fewer-colors`(色を増やすとダサい)。
  - `akin` — **関連が深い**話（対称）。例：`proximity`↔`belonging`（ともに近接の原理）。
  - `tension` — **逆から見た／対になる**話（対称）。例：`soft-contrast`(強すぎ回避)↔`ensure-contrast`(弱すぎ回避)。
  - `then` — **前後で効いてくる**（向きあり）。書いた側→`to` が「この次に」、`to` から見ると「その前に」。例：`rough-first`→`show-early`。
- 定義・描画は `index.html` の `REL` / `tiesFor()` / `tiesBlock()`。種別を増減したら `REL` と `REL_ORDER` を直す。

## 更新履歴は毎回追記する（必須）

サイトに見える変更（Tips の追加・削除、図の修正、UI・挙動の変更など）を加えたら、**その作業のなかで必ず `history.js` の `HISTORY` 配列に1項目追記する。** 履歴の更新は後回しにしない・忘れない。（ファイル分割や内部リファクタなど、利用者に見えない変更は履歴に載せない。）

- `HISTORY` は **新しいものを上**に積む。今日の日付で1ブロック足すだけ（既存ブロックの書式に合わせる）。
- 文面はサイトの声で、利用者に伝わる言葉で書く（内部の実装用語ではなく「何がどう良くなったか」）。Tips を増やしたときは追加したタイトルを挙げる。
- コミットやPRを分けても、履歴への追記は同じ変更とセットで行う。「変更したのに履歴に載っていない」状態を作らない。

## 仕掛かり中の作業は `PROGRESS.md` に書く

セッションを跨いだり、途中で使用量制限にぶつかっても作業を継続できるよう、**完了しきっていない作業がある間は `PROGRESS.md` に状況を残す。** これは作業メモであり、サイトには出ない（`history.js` とは別物。`PROGRESS.md` は「まだ終わっていないこと」、`history.js` は「終わって利用者に見える変更」）。

- 作業を中断する前・節目ごとに、以下を `PROGRESS.md` に最新化する：**いま何をしているか／どこまで終わったか／次にやること／触っているファイル／注意点や保留中の判断**。
- 新しいセッションを始めたら、まず `PROGRESS.md` に目を通して続きから入る。
- その作業が完了したら、該当項目を `PROGRESS.md` から消す（必要なら見える変更を `history.js` に追記する）。仕掛かりが何も無ければ `PROGRESS.md` は空でよい。
- `PROGRESS.md` は git 管理に含め、コミットして状態を持ち越せるようにする。

## 図を作るときの最小チェックリスト

作図ルール全文を読んだ上で、最低限これを満たす：

- [ ] SVG は `viewBox="0 0 320 120"`（縦積みのみ160まで）
- [ ] 外周16の余白。端に寄せない
- [ ] 色は CSS 変数のみ。生hex禁止
- [ ] accent は1図に1箇所まで
- [ ] Before/After は 左=問題（くすませる）右=解決（calm、注目点だけaccent）
- [ ] 矢印は三角（角つき線矢印は禁止）
- [ ] 棒グラフはゼロ始点、円グラフは2カテゴリまで
- [ ] `role="img"` と一文の `aria-label` がある
- [ ] 1図1主張。盛らない

## 種の工房（`tane.html`）— 図とネタの設計 → プロンプト化

作者が**図のラフとネタを直感的に置いて設計し、それを「Claude Code に渡せば実装される作図プロンプト」に変換する**単体ページ。`index.html` と同じ階層に置き、ビルドなしで動く。サイト本体からはリンクせず、`README.md` からのみ辿れる作者向けの工房。

- **役割分担**：工房は作図しない。ラフ（たたき台SVG）と意図の言語化までを担い、**規約準拠の最終 `visual` SVG は、プロンプトを受け取った Claude Code が `zu-drawing-rules-prompt.md` に従って清書する**。
- **構成**：①ネタ入力（このリポジトリのスキーマに対応：`cat`/`tags`/`id`/`title`/`claim`/`why`/`apply`/`term`）②ラフ作図キャンバス（`viewBox 0 0 320 120`・外周16ガイド／ブロック・文字・三角矢印・棒・丸をドラッグ配置／役割色を意味で選択／accentが2つ以上で警告）③プロンプト生成＋コピー。
- **CATEGORIES とデザイントークンは `index.html` と同値を複製している**（単体配信のため）。カテゴリの増減や CSS 変数を変えたら、`tane.html` 側も合わせること。プレビューも生hex禁止・CSS変数で色を解決する。
- **生成プロンプトの中身**：スキーマ下書きオブジェクト＋ラフSVG＋意図の箇条書き。実装手順として「規約準拠SVGに清書」「`tips.js` の `TIPS_<cat>` に1ブロック追記」「`history.js` に当日分を追記」「最小チェックリストで自己検算」まで指示する。**この工房の出力を受け取ったら、その手順どおりに実装する。**
- このツール自体は作者向けで来訪者向けではないため、`history.js`（サイトの更新履歴）には載せない。

- 既知の許容例外：`color` / `draw` の「多色は散らかる」を示す **Before（悪い例）パネルに限り**、多色見本のため生hexを使っている（After とそれ以外の全図は CSS 変数のみ）。
